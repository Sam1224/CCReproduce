## 方法概述 (中文)

确保大语言模型输出合规、安全需要实时内容审核，但现有护栏模型（如 LlamaGuard 系列）依赖 7B–27B 参数的自回归解码器，将本质上是分类问题的任务重新表述为顺序文本生成，导致高延迟和多维度评估时的效率瓶颈。

**GLiGuard** 将 LLM 内容安全护栏重新设计为模式条件（Schema-Conditioned）分类问题：

1. **架构基础**：以 GLiNER2（双向编码器，0.3B 参数）为骨干，适配为多方面分类器
2. **Schema 条件化**：以安全维度（prompt 安全性、response 安全性、拒绝检测、14 类细粒度伤害类别、11 类越狱策略）作为 schema 输入，单次前向传播同时评估所有维度
3. **并行多任务分类**：双向注意力机制捕获长程和末段伤害信号，避免自回归逐 token 生成的顺序成本
4. **Schema 组合灵活性**：支持任意安全维度组合，无需重新设计 prompt

## 故事线 (Story Arc)

> **现状不足：** LLM 护栏模型（7B–27B）将分类问题包装为生成问题，延迟极高（最高 17 倍于编码器），吞吐量低（最高 16 倍差距），难以支持实时内容审核场景；多维度同时评估需多次调用，进一步加剧延迟。
>
> **我们的解法：** GLiGuard 用 0.3B 双向编码器 + 模式条件化设计，将护栏还原为分类本质，在 9 个标准安全基准上达到竞争性 F1，同时实现 16× 吞吐量提升和 17× 延迟降低。

## 创新点分析

| 维度 | 描述 |
|------|------|
| 核心创新 | 首次将 GLiNER 双向编码器范式引入 LLM 安全护栏；schema 条件化单次多维度分类 |
| vs. 先前工作 | LlamaGuard/ShieldLM 等均为自回归解码器；GLiGuard 逆转范式，效率优先 |
| 可行性 | 9 个基准上验证；F1 与 7B–27B 模型竞争，尺寸缩小 23–90 倍 |
| 局限 | 编码器在推理中缺乏链式思维能力，对高度模糊场景解释性较弱 |
